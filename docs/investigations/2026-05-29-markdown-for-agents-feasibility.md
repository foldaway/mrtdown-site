# Markdown Alternatives for Agents Feasibility

Investigated whether mrtdown can expose Markdown alternatives for agents and
scrapers, in the style of Cloudflare documentation pages.

## External Patterns

- The `llms.txt` proposal recommends a root `/llms.txt` Markdown file with a
  concise site summary and links to detailed Markdown resources. It also
  proposes clean Markdown versions of useful pages at the same URL with `.md`
  appended, with URL-directory pages using `index.html.md`.
- Cloudflare's current productized feature is primarily HTTP content
  negotiation: clients send `Accept: text/markdown` and Cloudflare converts
  eligible HTML responses into Markdown at the edge. Cloudflare docs also
  expose explicit `index.md` links and `llms.txt` indexes.
- Cloudflare's feature currently converts only HTML, has a 2 MB origin response
  limit, and is available on Pro, Business, Enterprise, and SSL for SaaS zones.

## Repository Fit

mrtdown is a good fit for app-owned Markdown output:

- The app already runs on Cloudflare Workers through `app/server.ts`.
- File-based server routes already return non-HTML responses, for example
  `app/routes/{-$lang}/sitemap[.]xml.tsx`.
- Route loaders already call reusable data functions such as
  `getOverviewData`, `getLineProfileData`, `getStationProfileData`,
  `getOperatorProfileData`, `getIssueData`, and `getStatisticsData`.
- The route tree can express literal dots with the existing `[.]` convention,
  so Markdown routes should fit the current TanStack Start routing model.
- `public/robots.txt` and the dynamic sitemap give us existing discovery
  surfaces where `/llms.txt` or Markdown alternates could be linked.

## Decision

Build app-owned Markdown routes and a curated `/llms.txt` entry point.

This is the preferred path because it produces predictable agent-oriented
content from the same read-model data as the HTML pages, without depending on
HTML-to-Markdown conversion quality.

## Initial Scope

Ship a narrow English-first surface:

- `/llms.txt`: curated agent entry point.
- `/index.md`: current system status and links to line/status pages.
- `/lines/$lineId/index.md`: line profile, current status, recent issues,
  upcoming maintenance, interchange stations, and summary stats.
- `/stations/$stationId/index.md`: station profile, served lines, current
  status, recent issues, and nearby metadata.
- `/operators/$operatorId/index.md`: operator profile and line performance.
- `/issues/$issueId/index.md`: issue summary, affected lines/stations,
  intervals, updates, sources, and status.

Defer `/statistics/index.md`, history Markdown, full-site dumps, and localized
Markdown until traffic or use cases justify the extra surface area.

## URL Shape

Use directory `index.md` URLs as canonical:

- `/index.md`
- `/lines/EWL/index.md`
- `/stations/CG2/index.md`
- `/issues/2025-01-01-sample/index.md`

This fits the existing nested route shape and mirrors Cloudflare's explicit
docs links closely enough for agents.

Do not add `.md` aliases such as `/lines/EWL.md` in the first pass. Add
redirects or duplicate aliases later only if analytics show meaningful requests
for that shape.

## Content Policy

Default to `en-SG` only. Localized Markdown can come later after the base format
has settled.

Include public crowd-sourced report signals where the corresponding HTML page
would show them. They are high-value operational context for agents. Keep the
same feature flag and public cache behavior used by the page data so Markdown
does not expose a different product surface.

Do not add `Content-Signal` in the first pass. The routes should be usable as
agent input and search snippets, but training-policy semantics should be a
separate product/legal decision rather than an incidental header choice.

## Implementation Notes

- Put formatting in a shared utility such as `app/util/markdown.functions.ts`
  or `app/util/agentMarkdown.ts`, not inside React components.
- Use small Markdown escaping helpers for headings, links, tables, and fenced
  values. Avoid serializing rendered HTML.
- Set `Cache-Control: public, max-age=0, s-maxage=60,
  stale-while-revalidate=300` or reuse a generalized variant of the public HTML
  cache policy.
- Add `Vary: Accept` only for content-negotiated routes. Explicit `.md` routes
  do not need it.
- Keep generated route-tree updates generated by TanStack tooling.

## Deferred Options

### Cloudflare Markdown for Agents

Cloudflare's feature can still be enabled later as a fallback for clients that
send `Accept: text/markdown`, but it should not be the source of truth for
explicit `*.md` URLs.

Reasons to defer:

- It does not by itself create explicit `*.md` URLs.
- Output quality depends on rendered HTML, including navigation and app shell
  markup.
- Cloudflare's 2 MB origin-response limit needs monitoring on heavier pages.
- Dashboard availability and behavior are Cloudflare-plan dependent rather than
  fully controlled in this repo.

### Static Markdown Artifacts

The canonical data pull workflow could eventually pre-render Markdown files to
static assets or a KV/R2-like store. Defer this until Markdown traffic is heavy,
request-time rendering becomes a measurable problem, or a full-site Markdown
corpus is needed.

## Suggested Rollout

1. Add a small Markdown serialization utility and unit tests.
2. Add `/llms.txt` with curated links to the highest-value Markdown routes.
3. Add home, line, station, operator, and issue Markdown routes.
4. Add sitemap or robots discovery only after the routes are stable.
5. Log or inspect Cloudflare analytics for `text/markdown`, `.md`,
   `/llms.txt`, and bot user-agent traffic before expanding to history and full
   statistics.
